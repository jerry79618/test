# 需求文件：用戶身份驗證系統 (Auth Service v2.0)

## 1. 專案概述
本文件定義了 Auth Service 的核心身份驗證流程，包含登入、多因子驗證 (MFA) 以及帳號保護機制。目標是為 Java/Spring Boot 後端提供高安全性的認證邏輯。

## 2. 用戶角色
* **普通用戶 (User)**：可以進行登入、設定 MFA。
* **管理員 (Admin)**：具備解鎖帳號與查看登入日誌的權限。

## 3. 使用者故事 (User Stories)

### US-1: 基礎帳密登入
* **描述**：作為一個用戶，我可以使用帳號與密碼進行登入，以便存取受保護的資源。
* **驗收標準**：
    1. 密碼必須經過 Argon2 雜湊儲存。
    2. 登入成功後應回傳有效期為 60 分鐘的 JWT (Access Token) 與 Refresh Token。
    3. 登入失敗應回傳統一的錯誤訊息（防止帳號列舉攻擊）。

### US-2: 多因子驗證 (MFA)
* **描述**：為了提高安全性，在登入後若帳號已開啟 MFA，需強制進行二次驗證。
* **驗收標準**：
    1. 支援 TOTP (如 Google Authenticator) 或 SMS OTP。
    2. 登入後若檢測到 MFA 開啟，JWT 應標記為 `partial_auth` 狀態，僅限存取驗證 API。
    3. OTP 驗證碼有效期為 5 分鐘，且一經使用立即失效。

### US-3: 帳號鎖定與頻率限制 (Rate Limiting)
* **描述**：系統應防止暴力破解攻擊。
* **驗收標準**：
    1. 同一 IP 在 1 分鐘內嘗試超過 10 次登入，應暫時封鎖該 IP 5 分鐘。
    2. 同一帳號連續密碼錯誤 5 次，應將帳號鎖定 (Lockout)，需由 Admin 解鎖或等待 30 分鐘自動恢復。

## 4. 非功能性需求 (Non-Functional Requirements)
* **安全性**：必須遵循 Zero Trust 原則，所有 API 需強制 TLS 1.3 加密。
* **效能**：登入驗證回應時間 (Latency) 應低於 200ms（含 Argon2 計算時間）。
* **追蹤性**：所有失敗的登入嘗試必須記錄日誌 (Audit Logs)，包含 IP、時間與失敗原因。

## 5. 待釐清的潛在問題 (給 Agent 辯論的關鍵點)
* 當用戶同時開啟 SMS 與 TOTP 時，驗證優先順序為何？
* 若帳號處於「鎖定狀態」，是否仍應寄送「異地登入提醒」郵件？
* JWT Refresh Token 是否支援 Rotate 機制以防止竊取？
